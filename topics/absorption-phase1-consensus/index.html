Phase 1 でやろうとしたこと

Phase 0（板データ4指標、Book Depth視点）で「吸収シグナルは実用できそう」というところまで確認できた。 次はCVD（累積出来高デルタ）の加速度とOI（建玉）の変化も加えて、3つの視点で吸収を捉えよう——これがPhase 1の設計。

たった1つの考え方: 板だけでなく「誰がどの方向でポジションを持ってるか」も見れば、吸収の質を判断できるはず。

3視点の概要（VP/CVD + OI + Book）

視点1: VP/CVD（出来高の流れ）

指標なにを測るか優しい言い方
F1 CVD加速度net_tradeのゼロクロス「買い優勢↔売り優勢の切り替え」
F2 CRR_1s売 ÷ max(補充, 1)「1秒単位の板消費のキツさ」
F3 VNR出来高急増 + 値幅縮小「大量の注文がレンジ内で処理された」
F4 フェイクアウトリスクmicroprice乖離「ミッドと加重価格が離れすぎてる」

視点2: OI（建玉の変化）

指標なにを測るか優しい言い方
F5 OI-Price乖離OI方向 × 価格方向「逆張りしてる人がいるか」
F6 OIインパルスOI急変 + 値幅縮小「ポジション爆増 or 爆減」
F7 状態遷移OI・価格・CVDの3軸分類「買い優勢→売り優勢に切り替わった」

視点3: Book Depth（板の深さ）

Phase 0の4指標（CRR_5s / WPS / RVZ / TMR）をそのまま流用。

ゲート: G1 + G2

厳格コンセンサスがまったく動かなかった話

最初の設計はこうだった:
「3視点のうち2つが合意 かつ G1とG2の両方を通過 → +1シグナル」

しかし、結果は……

222,947行のデータに対して、+1シグナルは0件。何も出なかった。

🤔 なぜ0件？
「各視点の精度が悪い」と思ったが、本当の原因は別にあった。

どこで詰まったか — 件数の積集合

各条件の通過件数を1つずつ確認していくと、原因は一目瞭然だった:

2-of-3 consensus:      4,559件  ← 候補はある
G1通過（OI一致）:        429件  ← これでも可
G2通過（流動性OK）:  144,673件  ← 問題なし
G1 & G2:                 143件  ← ここで激減
consensus & G1 & G2:       1件  ← ほぼゼロ！
その1件も崩壊ラベルで+1にできない

問題の本質: 各条件が「正しい」かどうかは別として、疎な条件同士のANDで候補が消滅していた。 G1の通過率が0.2%と極端に低く、G2をかけるともう戻ってこない。

さらに悪化要因:

  • F7（状態遷移）が全件0 — 完全一致7連続は現実的に起こらない
  • OIの成立率が約1% — ボトルネックとして機能しすぎている
  • 正例が61/222,947（0.027%） — ラベルそのものが希少

GPT-5.5の診断「疎な条件のANDは死ぬ」

ここでGPT-5.5に相談。返ってきた答え:

主因は各ビューではなく、consensus & G1 & G2 の積集合が極端に疎なこと。
弱いモデル以前に、疎な条件同士のANDで候補を消滅させている。
strict consensusを最終発火条件として使う設計は放棄すべき。

GPT-5.5が提案した修正:

  1. G1/G2 の同時必須をやめる → 加点項目にする
  2. F7 をハードゲートから外す → 全件0の条件は意味がない
  3. OI も加点にする → 必須にすると候補が死ぬ
  4. 残すハードリジェクトは collapse だけ
新しい判定ロジック
candidate = 2-of-3 consensus かつ collapseでない
total_score = views_agree * 0.30
            + G1 * 0.15
            + G2 * 0.10
            + OI_continuous * 0.10
            + F7_bonus * 0.10
    
signal = candidate かつ total_score >= train_quantile(P98)

候補生成＋スコアリングに転換

書き換えたバックテストでは、strict consensus を最終発火条件から外した。 「まず候補を広く取って、その上でスコアでランク付けする」という、素直な二段構え。

閾値はパーセンタイルスイープで決める。P50からP99.5まで振って、件数・Precision・Recall・Forward Return を一覧する。

この記事で学んだこと

1️⃣ 疎なANDは思考停止する

G1通過率0.2% × G2通過率35% = 0.07%。さらに2-of-3に絞ると0.02%になる。 正例が0.027%の世界では、AND 3段は確実に候補を消す。「ANDは2まで」が安全域。

2️⃣ 全件0の特徴量は即刻外す

F7_transition（状態遷移検出）が全件0だった。完全一致7連続は現実的でない。 死んだ特徴量は判定材料として機能しない。「テストデータで0なら設計が間違っている」と割り切る。

3️⃣ 疎な指標はゲートにしない

OI（成立率約1%）を必須条件にすると、他の条件が満たせても救えない。 疎な指標は加点に回す。必須にするのは通過率50%以上の指標だけ。

4️⃣ パーセンタイル閾値で絞るのが安全

固定閾値（0.60、0.35等）はデータ分布が変わると意味を失う。 学習期間のパーセンタイルで「上位N%だけ発火」にすると、データ分布に適応する絞り込みができる。

次にやること

  1. F7の再設計: 完全一致7連続 → 直近の状態遷移有無（より現実的な検出ロジック）
  2. OI特徴量の調整: 通過率1% → 3-5%に緩和して加点として使えるようにする
  3. Walk-forward検証: パーセンタイル閾値をローリングで適用して過学習をチェック
  4. GPT-5.5 + code-reviewer 95点ゲート: 設計確定後にレビューを通過させる
